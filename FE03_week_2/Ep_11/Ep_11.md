### Whats the difference between state and prop

- state is local variable
- props is the arguement
  - its used to pass data from one component to another component

### Props drilling

- suppose you have state variable in a top most level component
  - and you want to pass those state variables to bottom most level components, how do you do it?
    - one of the ways would be to, passing down the props from one component to another to another in hierarchy format, this is prop drilling

- what if you want to send data from child to parent component?

- one of the disadvantages of prop dilling
  - is that you have to pass down props to all the components in between top lvl and bottom lvl component, if passing to bottom lvl component

### Lifting the state up

- suppose you have 3 sections which have text, with buttons show and hide, like accordion
- each section has a state, which allows the show and hide feature
- now you want to implement feature that, whenever user clicks on "show" on one of the section, you want other sections to close down, for that you need to put states in each of the section to its parent component, that is lifting the state up

- In cases where we need data across throughout our app, we have to store it in central space
- one of the options is local storage, but local storage is not the option, as it is a costly operation, and heavy operation
- react ka they have central space which is known as context, also there is redux state, there are also other state libraries like ngrx
- why to use context?
  - so that we could access the state anywhere across the app

### CreateContext

- createContext is a function, and it takes in some data that we need across the application
- initially it will take default value of the context
- we can create multiple context

- normal props and useState and all, they are tied to a component, context are not tied to component
- We use context only for situations where we need the "data" across all the project in different places
- useContext is like useState but for your whole big application
- we can use context wherever we wish to

- userContext(created from createContext) has a method called provider
  - e.g
    <UserContext.Provider value={"abcd"}>
    </UserContext.Provider>
- isse, we can override the default value
- The .Provider method is usually used in cases like App.tsx
